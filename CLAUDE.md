# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

MDEdit is a WPF desktop Markdown editor (.NET, `net10.0-windows`) built around the AvalonEdit text editor control. See `Requirements.md` for the full product spec (document management, editing, Markdown formatting commands, syntax highlighting, keyboard shortcuts, toolbar, status bar). A File-menu MRU (recent files) list and an editor font picker are spec'd in Requirements.md §1/§6 but marked *(planned)* — not yet implemented, not a bug.

WYSIWYG is under active, incremental implementation (see Architecture below) as a **live-preview mode inside AvalonEdit itself** — not a WebView2/rendered-HTML view. The Markdown text is always the single source of truth; the toggle (View → Editor Mode → WYSIWYG) only changes how AvalonEdit *renders* that same text (larger heading fonts, hidden syntax markers off the caret's line, eventually hidden bold/italic/link markers too). This was a deliberate choice over a `contenteditable`/WebView2 approach specifically to avoid a lossy HTML→Markdown round-trip — there is no round-trip, so there's nothing to lose. A WebView2 *read-only* rendered preview pane remains a possible separate future addition, not a replacement for this. Currently implemented: heading font scaling + marker hiding (per-line reveal), bold/italic/bold+italic/strikethrough/inline-code marker hiding (per-span reveal — only the run the caret is inside stays visible, not the whole line; see `EmphasisMarkerElementGenerator`), fenced-code-block fence-line hiding (per-block reveal — the whole ```` ``` ````/`~~~` line hides unless the caret is anywhere inside that block, not just on the fence line itself; see `CodeBlockFenceElementGenerator`, the first *multi-line* construct here), link/image marker hiding (per-span reveal like emphasis, but with an asymmetric prefix/suffix instead of a symmetric marker — see `LinkMarkerElementGenerator`), and blockquote marker hiding (per-line reveal like headings — see `BlockquoteMarkerElementGenerator`). Not yet done: lists — extend the same mechanism (see `Editing/` below) incrementally, one construct at a time.

## Build & Run

```
dotnet build MDEdit.slnx
dotnet run --project MDEdit/MDEdit.csproj
dotnet test MDEdit.slnx
dotnet test MDEdit.slnx --filter "FullyQualifiedName~MarkdownFormatterTests"   # single class; ~MethodName works too
```

`MDEdit.Tests` (xUnit, `net10.0-windows` — it needs the WPF-targeted AvalonEdit assembly) covers the logic pulled out of `MainWindow.xaml.cs` into `MDEdit/Editing/` specifically so it's testable without a running window: `MarkdownFormatter` (formatting commands) and `MarkdownSyntax` (line-construct detection). `MDEdit.csproj` grants it `InternalsVisibleTo` so those `internal` classes are testable directly. Running `dotnet test` is fine to do yourself; do not launch/drive the actual GUI app to verify behavior — that's for the user to do and report back.

Release builds (`dotnet build -c Release` / `dotnet publish -c Release`) are Authenticode-signed automatically via an `AfterTargets="Build"` MSBuild target in `MDEdit.csproj` that runs `build/Sign.ps1`. It signs with whatever cert matches subject `CN=Maze Code Signing` in the CurrentUser store (matched by subject, not thumbprint, since the cert rotates yearly) and timestamps against `http://timestamp.digicert.com`. This requires network access and the cert being present — Release builds will fail without them. Debug builds are unaffected.

### ClickOnce deployment

`docs/` is a live GitHub Pages site (`https://cowwarrior.github.io/MDEdit/`) serving a ClickOnce deployment — this is why the repo is public rather than private. To publish a new version, run `build/Publish-ClickOnce.ps1`, then commit and push the changes under `docs/`.

- The script must be run standalone (not via `dotnet publish`): ClickOnce's `UpdateManifest` task only runs under the full-framework MSBuild bundled with Visual Studio (`MSBuild.exe`, located dynamically), not the cross-platform MSBuild `dotnet` uses (fails with MSB4803 otherwise).
- ClickOnce manifest signing (the separate XML-DSig signature over `MDEdit.application`/`*.manifest`, distinct from the Authenticode signing above) requires an RSA certificate — the same `CN=Maze Code Signing` cert is used, resolved by subject at publish time, never by thumbprint (it rotates).
- The publish script derives `ApplicationVersion` as `1.0.0.<git rev-list --count HEAD>` and passes the full 4-part version directly — **not** via the separate `ApplicationRevision` property. This MSBuild's `FormatVersion` task (`Microsoft.Build.Tasks.Core.dll`, VS 18) silently ignores `Revision` regardless of how many parts `ApplicationVersion` has, so every publish would otherwise produce the identical version `1.0.0.0` and installed clients would never see an update (verified empirically — confirmed and fixed after finding this in practice).
- The deploy is framework-dependent (`SelfContained=false`) — self-contained ClickOnce output is ~350MB, impractical to version in git. This means machines installing MDEdit need the [.NET 10 Desktop Runtime](https://dotnet.microsoft.com/download/dotnet/10.0) already present; there's no bootstrapper to auto-install it.
- `MDEdit/Properties/PublishProfiles/ClickOnce.pubxml` holds the static settings (URLs, product/publisher name, `SelfContained=false`, etc). It deliberately omits `ManifestCertificateThumbprint` and `ApplicationRevision` — the script supplies both at publish time.
- The script also deletes any `Application Files\MDEdit_*` folder under `docs/` other than the one it just published — ClickOnce publish only ever adds a version folder, never removes superseded ones.
- `.gitattributes` marks everything under `docs/` as binary (`-text`) except `*.html` — git's CRLF normalization would otherwise corrupt the byte-exact hashes ClickOnce embeds in the manifest.
- `docs/index.html` is the hand-maintained install landing page (the only file under `docs/` not regenerated by publish); it links to `MDEdit.application` and notes the .NET 10 Desktop Runtime prerequisite — update it by hand if install instructions change.
- **The pubxml's `FileAssociation` items (`.md`/`.markdown`/`.txt`) do nothing** — verified empirically (no registry keys or event-log activity at all after a fresh install) that ClickOnce's file-association registration was never wired up for Launcher-based .NET Core deployments, despite the tooling accepting the declaration without error. Real file association is handled entirely by `Services/FileAssociationService.cs` instead (see Architecture below); the pubxml entries are dead config left in place mostly as a documented dead end, not because anything depends on them.

## Architecture

The app is a single-window WPF application; almost all logic lives directly in `MainWindow.xaml.cs` rather than being split across MVVM layers — there is no ViewModel.

- **`MainWindow.xaml` / `MainWindow.xaml.cs`** — the entire UI (menu, toolbar, status bar, editor) and its code-behind. Command routing uses standard WPF `RoutedUICommand`s (`ApplicationCommands.New/Open/Save/...`) plus custom ones declared as `static readonly RoutedUICommand` fields (`SaveAsCommand`, `BoldCommand`, `ItalicCommand`) for actions with no WPF built-in. Heading shortcuts (Ctrl+1/2/3) are wired via `InputBindings` + a local `RelayCommand` rather than `CommandBinding`, since there's no built-in command for them.
  - Formatting operations (bold/italic/strikethrough/code = `WrapSelection`; headings = `InsertHeading`; lists/blockquote = `InsertLinePrefix`; code block = `InsertCodeBlock`; links = `InsertLink`) all follow the same convention described in Requirements.md §3: wrap the selection if there is one, otherwise insert placeholder syntax and position the caret/selection for immediate typing. These are now thin wrappers around `Editing/MarkdownFormatter.cs` (see below); MainWindow just supplies the `TextDocument` + current selection and applies the returned caret/selection placement via `ApplyFormat`, which sets it with a single `Editor.Select(start, length)` call — not separate `SelectionStart`/`SelectionLength` assignments, since `TextEditor.SelectionStart`'s setter internally reuses whatever `SelectionLength` currently reports, which can be a stale, now-out-of-bounds value left over from AvalonEdit's selection-anchor bookkeeping after `MarkdownFormatter`'s `doc.Replace` (reproducible by wrapping a selection that reaches end-of-document; caused a real `ArgumentOutOfRangeException` crash, fixed and covered by `ApplyFormatSelectionTests`).
  - Live preview / WYSIWYG state: `_settings.LivePreview` (persisted) drives `UpdateLivePreviewState()`, which flips `MarkdownLineColorizer.LivePreviewEnabled`, `HeadingMarkerElementGenerator.Enabled`, `EmphasisMarkerElementGenerator.Enabled`, `CodeBlockFenceElementGenerator.Enabled`, `LinkMarkerElementGenerator.Enabled`, and `BlockquoteMarkerElementGenerator.Enabled`, and syncs the View → Editor Mode radio-style menu checkmarks (`MenuEditorModeSource`/`MenuEditorModeWysiwyg`, set manually like the Theme submenu — no native radio-group support used elsewhere in this codebase either). `OnCaretPositionChanged` tracks both `_lastCaretLine` (heading, code-fence, and blockquote reveal, line-grained) and `_lastCaretOffset` (emphasis and link reveal, span-grained — a caret move within the same line can still move into/out of a bold/italic/link run) and redraws just the old and new line via `TextView.Redraw(DocumentLine)` whenever the offset changes at all; generator state is updated *before* those redraws so both lines render against the new caret position. Code-block fences are the one exception to "just the old/new caret line": a fence pair can sit many lines away from the caret's line (the only multi-line-scoped construct so far), so `OnCaretPositionChanged` additionally calls `RedrawEnclosingFenceLines()` for both the old and new caret line, which looks up `MarkdownSyntax.TryGetEnclosingFenceBlock` and redraws that block's fence lines specifically — moving the caret into or out of a code block has to reveal/hide fences that may be well outside the two lines already being redrawn. Blockquotes deliberately do *not* get this block-wide treatment despite spanning multiple lines the same way fences do — only the caret's own line reveals its `>` marker, matching the simpler per-line pattern. `ResetLivePreviewCaretTracking()` re-syncs all five generators after `NewDocument`/`OpenFile` reset the caret to line 1 (called wherever `UpdateHighlighting` is, for the same "just loaded a different document" reason).
  - Document dirty-state (`_isDirty`), title bar, and status bar text are kept in sync through `MarkDirty()` / `UpdateTitle()` / `UpdateStatusBar()`, called after edits, caret moves, and file operations.
  - `CheckUnsavedChanges()` is the single choke point for the "save/discard/cancel" prompt required before New, Open, and window Close.
  - The constructor checks `Environment.GetCommandLineArgs()` for a file path (`args[1]`) and opens it via `OpenFile(path)` — needed because `FileAssociationService` launches MDEdit as `"MDEdit.exe" "%1"`; without this, double-clicking an associated file opened an empty window.
  - `ApplySettings()` applies loaded `AppSettings` (word wrap, line numbers) on startup; the corresponding menu click handlers save back to `SettingsService` immediately on toggle, not just on close.
- **`AboutWindow.xaml` / `.xaml.cs`** — small owned dialog for Help → About. Shows `Resources/AppIcon.png` (not the `.ico` — WPF's `Image` control renders a multi-res `.ico` from its smallest embedded frame and upscales it, which looks blurry above ~32px), plus the assembly version and build date read via reflection (`AssemblyMetadataAttribute` with key `"BuildDate"`, stamped by the `SetVersionFromGit`/`AssemblyMetadata` bits in `MDEdit.csproj` — see below).
- **`Services/FileService.cs`** — tracks `CurrentPath` and does the actual disk I/O (`LoadFile`, `Save`, `SaveAs`, `Reset`). Always reads/writes as UTF-8.
- **`Services/FileAssociationService.cs`** — registers MDEdit as the default handler for `.md`/`.markdown` under `HKCU\Software\Classes`, triggered manually via Help → Register File Associations (not automatic). This exists because ClickOnce's own `FileAssociation` manifest feature is a no-op for this deployment (see ClickOnce section above). `.md`/`.markdown` are made the actual default handler; `.txt` is deliberately registered only as an "Open with" choice (`OpenWithProgids`), never as the default — it commonly already has a default handler, and taking over the default triggered an unwanted "change your default editor" prompt. This split is intentional and is exactly the desired behavior.
- **`Services/SettingsService.cs`** — persists `AppSettings` (`WordWrap`, `ShowLineNumbers`, `Theme`, `LivePreview`) as JSON at `%AppData%\MDEdit\settings.json`. Falls back to defaults on a missing/corrupt file rather than failing startup.
- **`Services/ThemeService.cs`** — implements the View → Theme options (Light/Dark/System). Two mechanisms in tandem: (1) sets `Application.ThemeMode` (WPF Fluent themes — an experimental API, hence `WPF0001` in `<NoWarn>` in the csproj) so standard controls, menus/popups, and dialogs restyle; (2) swaps which of `Themes/Light.xaml` / `Themes/Dark.xaml` is merged into app resources, driving the `DynamicResource` brushes MainWindow uses for the menu/toolbar/statusbar strips and the editor colors. "System" resolves via the `AppsUseLightTheme` registry value; MainWindow additionally listens to `SystemEvents.UserPreferenceChanged` to re-apply live when the OS theme flips (unhooked in `OnClosed`). The editor side is handled in `MainWindow.ApplyTheme()`: theme-aware `MarkdownLineColorizer` and `BlockquoteMarkerElementGenerator` brushes, caret brush, and a dark variant of the XSHD highlighting — built in `LoadDefinition(dark)` by recoloring the parsed XSHD model (`XshdColor` elements) before compiling it, since colors on a loaded `IHighlightingDefinition` may be frozen. Win32 `MessageBox`/common dialogs are not themed and stay light — known limitation, not a bug.
- **`MarkdownLineColorizer.cs`** — an AvalonEdit `DocumentColorizingTransformer` registered on `Editor.TextArea.TextView.LineTransformers`. Handles *line-level* styling that XSHD span/rule matching can't express well: heading color/weight by `#` count, blockquote italics (via `MarkdownSyntax.TryGetBlockquoteMarkerLength`, shared with `BlockquoteMarkerElementGenerator` so the two never disagree on what counts as a blockquote line), horizontal rules. This runs alongside (not instead of) the XSHD highlighting below, and none of it is gated by `LivePreviewEnabled` except the heading font scaling noted next — headings/blockquotes/hrules are colored this way regardless of editor mode. When `LivePreviewEnabled` is set, headings are additionally scaled up in `FontRenderingEmSize` per level (`HeadingScale`) — Typora-style larger heading text; this is purely a rendering effect via `ChangeLinePart`, the underlying document text is untouched.
- **`Editing/` folder** — UI-free classes factored out specifically to be unit-testable (see `MDEdit.Tests` above) and to anchor the live-preview implementation:
  - **`MarkdownFormatter.cs`** — the actual logic behind every formatting command (`Wrap`, `Heading`, `ToggleLinePrefix`, `CodeBlock`, `Link`), operating on an AvalonEdit `TextDocument` plus a plain `SelectionRange` struct rather than the live `TextEditor`, so tests can drive it without a window.
  - **`MarkdownSyntax.cs`** — shared line-construct detection: `TryGetHeadingLevel` (ATX heading + marker length), `TryGetBlockquoteMarkerLength` (leading `` `>` `` run, including nested `` `> > ` ``, plus a `depth` out param — the nesting level, used to draw one indent bar per level; see the `BlockquoteMarkerElementGenerator` bullet below), `FindEmphasisSpans` (bold/italic/bold+italic/strikethrough/inline-code runs on a line, as `EmphasisSpan { Start, End, MarkerLength }`; inline code's content is never recursed into — it's literal per CommonMark's code-over-emphasis precedence, which the lexer's leftmost-wins scan also provides at the top level, same as AvalonEdit's earliest-match rule resolution), `IsFenceDelimiterLine`/`TryGetEnclosingFenceBlock` (fenced-code-block detection — see the `CodeBlockFenceElementGenerator` bullet below), and `FindLinkSpans` (links/images, as `LinkSpan { Start, TextStart, TextEnd, End }` — see the `LinkMarkerElementGenerator` bullet below). `MarkdownLineColorizer` (coloring/sizing) and the marker-hiding generators both call into this rather than each having their own regex/parsing, specifically so they can never disagree about what counts as a given construct. `FindEmphasisSpans` mirrors `Markdown.xshd`'s exact patterns and precedence (bold+italic before bold before italic; character-class exclusion instead of wildcard quantifiers, to avoid catastrophic backtracking — same reasoning as the XSHD comments) via a tiny `\G`-anchored, non-overlapping single-pass lexer, so the two never drift apart. It recurses one level into each match's inner content to find a **mixed-delimiter nested run** (e.g. `_**bold**_`, `**_italic_**` — CommonMark forbids nesting the same delimiter inside itself, so this is the standard way authors combine bold+italic as two runs instead of `***text***`) — this falls out for free with no extra filtering, since a match's content can never contain its own delimiter (excluded by the pattern's own content class), so re-scanning that content with the same pattern list can only ever find an other-delimiter run; the same mechanism covers strikethrough wrapping/wrapped-by emphasis (`~~**text**~~`, `**~~text~~**`). `TryGetEnclosingFenceBlock` is the one method in this file that isn't purely line-local: fence pairing (which same-kind `` ``` ``/`~~~` line closes which) is a document-wide parity property, so it walks from the start of the document — callers check the O(1) `IsFenceDelimiterLine` first so this cost is only paid for actual fence lines, not the majority of lines in a document. `FindEmphasisSpans` and `FindLinkSpans` are mutually skip-aware — each tries the other's pattern table first at every scan position (without recording a match, just to advance past it) — mirroring how `Markdown.xshd`'s Link rule is a flat `<Rule>` with no nested `RuleSet` (unlike Bold/Italic's `<Span>`): `[**bold**](url)` is one opaque link with no separately-colored nested Bold, and `**[link](url)**` is one opaque Bold run with no separately-colored nested Link, in both the real syntax highlighter and here. Extend this file first when adding live-preview support for another construct (lists) — colorizer and generator(s) should both consume it.
  - **`HeadingMarkerElementGenerator.cs`** — an AvalonEdit `VisualLineElementGenerator` (registered on `Editor.TextArea.TextView.ElementGenerators`) that hides a heading's `"# "` marker text using a zero-size `InlineObjectElement`, on every line except the one the caret is currently on (**per-line** reveal — a heading marker only ever exists at a line's start, so line-granularity is sufficient). The marker characters still occupy their document offsets (selection/undo/the saved file are unaffected) — only the visual rendering collapses them to nothing.
  - **`EmphasisMarkerElementGenerator.cs`** — same zero-size-`InlineObjectElement` hiding technique, but **per-span** reveal instead of per-line: several bold/italic runs can share one line, so only the specific run the caret is inside (inclusive of both edges, so landing exactly on a boundary doesn't flicker) stays visible; hiding every marker on the line just because the caret is elsewhere on it would be distracting mid-sentence. This was the template for `LinkMarkerElementGenerator`.
  - **`BlockquoteMarkerElementGenerator.cs`** — same per-line reveal convention as `HeadingMarkerElementGenerator` (only the caret's own line shows its `` `>` `` marker, even for a multi-line blockquote — deliberately *not* the fence pair's per-block reveal, since a blockquote marker's meaning and `MarkdownLineColorizer`'s italic styling are entirely local to its own line), but the **replacement isn't zero-size** like every other marker generator: the hidden marker is replaced with an indent plus one left accent bar per nesting level (`MarkdownSyntax.TryGetBlockquoteMarkerLength`'s `depth` out param), so a blockquote reads as indented, bordered prose rather than "text that lost its `>`". `InlineObjectRun` measures its element with an *infinite* constraint (`TextView.cs`'s `AddInlineObject`/`InvalidateVisualLines` call `Element.Measure(new Size(double.PositiveInfinity, double.PositiveInfinity))`), so the bar can't rely on stretching to the line's height the way in-flow content does — its height is computed from `CurrentContext.GlobalTextRunProperties.FontRenderingEmSize` instead, so it scales with the editor's font size. List markers should follow the plain zero-size `HeadingMarkerElementGenerator` pattern when added — this indent-bar approach is blockquote-specific, not a new general template.
  - **`CodeBlockFenceElementGenerator.cs`** — same zero-size-`InlineObjectElement` technique again, but on the **whole line** (a fence line like `` ```csharp `` has no separate "marker prefix" vs. "content" the way a heading does — the entire line *is* the marker) and with **per-block** reveal: both the opening and closing fence of a code block stay visible whenever the caret is anywhere inside that block (on either fence line or any content line between), not just when the caret is literally on a fence line — otherwise the fences would flicker away while simply typing code in the middle of the block. This is the first live-preview construct where what's hidden and what triggers a redraw aren't confined to the caret's own line, since a fence pair can be many lines apart; see the `MainWindow.OnCaretPositionChanged`/`RedrawEnclosingFenceLines` note above.
  - **`LinkMarkerElementGenerator.cs`** — `EmphasisMarkerElementGenerator`'s per-span reveal convention applied to links/images, but with an **asymmetric** hidden region instead of a symmetric marker: `[` (or `![`) as the prefix and `](url)`/`][ref]` as the suffix, since the URL/reference portion has no fixed width the way `**`/`_` do — so this constructs two independently-sized `InlineObjectElement`s per link (`LinkSpan.TextStart - Start` and `LinkSpan.End - TextEnd`) rather than reusing one `MarkerLength` at both ends.
- **`Resources/Markdown.xshd`** — AvalonEdit XSHD grammar for *inline* Markdown highlighting (bold/italic/strikethrough/inline code/links/images/fenced code blocks/HTML comments/list markers). Embedded as a resource (see `.csproj`) and loaded at runtime via `Assembly.GetManifestResourceStream("MDEdit.Resources.Markdown.xshd")` in `LoadSyntaxHighlighting()`. Comments in the file note deliberate regex constraints (e.g. "no wildcard quantifiers", "min 2 chars") — these were tuned to avoid catastrophic backtracking / crashes in AvalonEdit's regex engine, so preserve that style when editing rules (see commit "Fix syntax highlighting crashes and rearchitect line-level highlighting").
  - Bold and Italic are `<Span>` elements (not flat `<Rule>`s) specifically so a nested nested `<Rule>` inside each can recognize the *opposite* delimiter family wrapped inside it (`**_text_**`, `_**text**_`) and color it `BoldItalic` explicitly (not `Italic`/`Bold` alone — that would risk the nested match's color overriding, rather than combining with, the outer span's weight/style). Flat same-family nesting (`**text**` inside `**text**`) is still unsupported and out of scope — CommonMark itself requires mixing delimiter families to nest emphasis unambiguously, so this covers the real cases. `MarkdownXshdTests.MarkdownXshd_LoadsWithoutError` in `MDEdit.Tests` loads this file through AvalonEdit's real `HighlightingLoader` (not just a build-time XML check) so a malformed grammar fails `dotnet test` rather than only surfacing when a `.md` file is opened.
  - Bold/Italic/emphasis matching is single-line only — `<Rule>` elements (and the content-class patterns above) cannot cross line breaks; only `<Span>` supports `multiline="true"` (used for fenced code blocks/HTML comments). Making bold/italic span multiple lines would need converting them to multiline Spans and making `MarkdownLineColorizer`/the marker-hiding generators aware of spans that start on one line and end on another (they're currently strictly per-`DocumentLine`) — a materially bigger change than the mixed-nesting fix above, not yet done.
  - Highlighting is swapped based on file extension in `UpdateHighlighting(path)`: `.md`/`.markdown` get the Markdown XSHD, anything else (including `.txt`) gets no highlighting, per Requirements.md §4.
- **`Resources/AppIcon.ico` / `AppIcon.png`** — both are `<Resource>` items (WPF pack-URI-resolvable, distinct from the `EmbeddedResource` used for the XSHD grammar). The `.ico` (multi-res, generated from the `.png`) drives `<ApplicationIcon>` in the csproj and `MainWindow`'s titlebar icon; the `.png` is used only in `AboutWindow` for the reason noted above.
- **`MDEdit.csproj`** build-time targets worth knowing about beyond the Authenticode one above: `SetVersionFromGit` sets `<Version>` to the same `1.0.0.<git commit count>` scheme the ClickOnce publish script uses (so the About dialog's version matches the deployed ClickOnce version), and a plain `<AssemblyMetadata Include="BuildDate">` item stamps the current UTC time into the assembly on every build — needed because deterministic builds (the SDK default) zero out the PE header timestamp that used to serve this purpose.

## Git workflow

- Before running `git commit` (and before `git push`), show the proposed commit message and wait for explicit confirmation — do not commit or push on the same turn as drafting the message without that check-in. Exception: once the user has given an explicit go-ahead (e.g. supplied the commit message themselves, or said "commit and push"), that covers the rest of that task, including any incidental fixes needed to actually complete it (e.g. a build error hit mid-publish) — don't stop to re-confirm those, just use a reasonable one-line message and proceed. Only a genuinely new/separate change needs fresh confirmation.
- Commit messages are one line only, unless the change is an extraordinary case that genuinely warrants a body.

## Conventions

- Nullable reference types are enabled; keep new code nullable-aware.
- File-scoped namespaces (`namespace MDEdit;`) are used throughout.
- Keep formatting-insertion behavior consistent with the "wrap selection / insert-and-position-caret" pattern already established for existing commands when adding new ones.
